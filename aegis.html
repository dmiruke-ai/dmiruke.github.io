AEGIS Certify

Deterministic AI compliance assurance — governance, evaluation, and safety layer for agentic AI systems.

PCUs: 115 across 12 frameworks Gates: G1–G17 (non-bypassable) Test cases: 866 Architecture: FAIL-dominant lattice Snapshot: 2026-05-19

01Overview

AEGIS Certify is a mathematically-grounded AI governance system that evaluates AI artifacts against regulatory predicates using executable Primary Compute Units (PCUs), enforced through a FAIL-dominant lattice gate architecture. The core design invariant: if any PCU fails, the gate fails — no compensation, no averaging, no overrides.

This is not a wrapper around an LLM judge. It is a deterministic evaluation engine: the same inputs always produce the same output. Policy decisions are auditable, reproducible, and traceable to specific predicate failures.

Problem

Agentic AI systems are ungoverned at runtime

LLM-based agents act on tools, call external APIs, spawn sub-agents, and take irreversible actions with no enforcement boundary between intent and execution.

Approach

Executable assurance via PCU lattice

Each action passes through an ordered gate sequence. Gates evaluate PCUs (discrete, testable predicates). Any FAIL propagates upward and halts execution — the lattice is FAIL-dominant by construction.

Coverage

12 regulatory frameworks, 115 PCUs

GDPR, EU AI Act, NIST AI RMF, SOC 2, HIPAA, PCI-DSS, Fairness, Safety, RAG, Monitoring, Data Governance, and 7 Agentic risk PCUs (R1–R7).

FAIL-dominant lattice

The lattice is a partial order on trust states: CERTIFIED > PROVISIONAL > FORBIDDEN. When multiple PCUs evaluate a gate, the gate's output is the meet of all PCU outputs — the least element in the lattice that dominates all inputs. A single FAIL drags the gate to FAIL regardless of how many PCUs pass.

Gate output = ⋀ { PCU₁, PCU₂, ..., PCUₙ }   (meet over FAIL-dominant lattice)

PASS ∧ PASS = PASS
PASS ∧ WARN = WARN
PASS ∧ FAIL = FAIL   ← no exception
FAIL ∧ FAIL = FAIL

Matroid independence for PCU composition

PCUs within a gate form an independence system: each PCU evaluates a distinct predicate (no two PCUs check the same regulatory clause). This guarantees that removing any subset of PCUs cannot create false positives — you can only reduce coverage, never inflate it.

06Context Graph — G16 Domain Drift Detection

Multi-agent workflows traverse a graph of nodes, each with a trust tier (CERTIFIED, PROVISIONAL, FORBIDDEN). Gate G16 enforces constraints on the transition path — not just individual nodes.

Node trust tiers

CERTIFIED

Fully validated

Trust tier 3. Agent may traverse without intervention. All PCUs pass for this node's domain and task type.

PROVISIONAL

Human-in-the-loop required

Trust tier 2. Enforcement: HITL. Path traversal triggers a hold for human review before execution continues.

FORBIDDEN

Halt on contact

Trust tier 0. Enforcement: HALT. Any path reaching a FORBIDDEN node fails immediately. Hard-mode flag makes this non-negotiable.

Example — healthcare agent path evaluation

Graph nodes:
  intake    → domain: healthcare, trust: 3 (CERTIFIED)
  diagnosis → domain: healthcare, trust: 3 (CERTIFIED)
  billing   → domain: finance,    trust: 2 (PROVISIONAL)
  external  → domain: internet,   trust: 0 (FORBIDDEN)

Path: [intake → diagnosis]
  → decision: PASS,  enforcement: NONE

Path: [intake → diagnosis → billing]
  → decision: WARN,  enforcement: HITL   ← domain drift: healthcare→finance

Path: [intake → diagnosis → external]  (hard_mode=true)
  → decision: FAIL,  enforcement: HALT   ← FORBIDDEN node contact

Domain drift (e.g. a healthcare agent suddenly calling a finance or internet node) triggers G16 even if the destination node is PROVISIONAL rather than FORBIDDEN. The context graph makes cross-domain transitions explicit and enforceable.

07Research Hypotheses & Experiments

Seven formal research hypotheses (H1–H7) drive the evaluation program. 866 test cases across 3 experiments. Results are reproducible via the live API.

H2 — Tested · 242 test cases
High Attack Detection Rate
"AEGIS blocks >80% of adversarial attack patterns without human intervention."
Result: 50.8% block rate in experiment run. Gap driven by novel jailbreak variants not yet covered by H3/H4 PCUs. Establishes baseline for PCU expansion.
H5 — Validated · 322 test cases
False Positive Analysis
"AEGIS maintains <5% false positive rate on legitimate task requests."
Result: 0% false positive rate across both experiment runs. Legitimate requests pass all 17 gates cleanly. Validates that FAIL-dominance does not over-block benign traffic.

08Live API & Interfaces

AEGIS Certify exposes four interfaces over the same assurance kernel. No installation required for REST/gRPC — the API is live.

REST API

FastAPI — Swagger UI

Interactive docs at /docs. Full OpenAPI spec at /openapi.json. Endpoints cover hypotheses, experiments, metrics, context-graph evaluation, and PCU registry.

gRPC

High-throughput evaluation

For latency-sensitive pipelines. Same gate sequence, same PCU lattice. Proto definitions in proto/aegis/.

Python SDK

Inline integration

Import aegis_certify directly into evaluation pipelines. RAGAS and DeepEval adapters for LLM eval workflows.

CLI

CI/CD integration

Gate evaluation as a pipeline step. Returns non-zero exit code on FAIL — blocks the build. OPA bundle export for policy-as-code workflows.

Quick API reference

# Health check
curl http://37.27.97.75:18000/health

# List research hypotheses (H1–H7)
curl http://37.27.97.75:18000/api/v1/hypotheses

# List available metrics
curl http://37.27.97.75:18000/api/v1/metrics/available

# Evaluate a context graph path
curl -X POST http://37.27.97.75:18000/api/v1/context-graph/evaluate \
  -H "Content-Type: application/json" \
  -d '{"graph_id": "", "path": ["intake", "diagnosis"]}'
# → {"decision": "PASS", "enforcement": "NONE", ...}

OPA & Kubernetes integration

PCU evaluation results export as OPA-compatible Rego bundles. This lets platform teams enforce AEGIS decisions at the Kubernetes admission webhook level — an agent workload that fails a PCU at deploy time is rejected before it reaches the scheduler. The same bundle integrates with Kyverno policy engine for declarative enforcement.
